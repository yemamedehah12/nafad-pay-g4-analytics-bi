1. Introduction & consignes générales

NAFAD-PAY est un système fictif de paiement mobile mauritanien. Vous travaillerez sur un jeu de données synthétique (seed 42) reproduisant les problématiques réelles d'un système de paiement distribué : concurrence, clock skew, idempotence, eventual consistency, réconciliation multi-nœuds.

Objectif pédagogique double Chaque groupe doit (1) implémenter une solution fonctionnelle sur les données fournies, ET (2) produire deux documents d'architecture : une version "Early Stage" (MVP livrable immédiatement) et une version "At Scale" (cible à forte charge). L'enjeu est de raisonner sur les trade-offs, pas seulement d'écrire du code.

Contexte métier

  • Monnaie : MRU (Ouguiya mauritanien)
  • Pays : Mauritanie - indicatif +222 - NNI 10 chiffres
  • 15 wilayas (régions), moughataas (départements)
  • Infrastructure : aucun cloud provider majeur n'a de datacenter en Mauritanie. Tous les groupes travaillent sur AWS (région cible eu-west-3 Paris, 3 AZ). Hetzner peut être évoqué en comparaison pour le bare-metal low-cost. À vous de justifier vos choix (latence, souveraineté, coût, conformité BCM).
Données synthétiques Les données sont générées avec seed 42 - aucun utilisateur réel. Elles contiennent des anomalies intentionnelles (doublons, soft-deletes, clock skew, conflits inter-nœuds). À vous de les détecter et d'en tenir compte dans vos schémas. Ne pas redistribuer hors du cadre de la formation.

Oubliez les datacenters fictifs à Nouakchott / Nouadhibou. Dans la vraie vie, il n'existe aucune région cloud majeure en Mauritanie ni dans les pays limitrophes. Vos seuls choix réalistes :

Contrainte structurante à intégrer dans vos docs d'archi Un aller-retour Nouakchott ↔ Paris coûte ~40-60 ms de latence réseau incompressible. Pour un système de paiement où chaque écriture passe par votre base, cela fixe un plafond dur sur le nombre de transactions qu'un utilisateur peut faire sans ressentir de lenteur. Votre archi doit en tenir compte (cache, edge, confirmation asynchrone, ou hébergement hybride VPS local + cloud).
Ce qu'on cherche à évaluer Un bon doc d'archi ne se limite pas à "mettre du AWS". Il justifie : choix de eu-west-3 vs af-south-1 (distance vs proximité africaine), multi-AZ vs multi-region, setup hybride VPS Hetzner local + AWS Paris pour la latence, stratégie de coût (Savings Plans, Reserved Instances), conformité BCM et exigences locales.

3. Template du document d'architecture

Utilisez ce plan pour les DEUX documents (Early Stage et At Scale). Même structure, contenu différent : le doc Early Stage décrit le MVP livré; le doc At Scale décrit la cible et comment y migrer.

3.1 Plan commun imposé

# Document d'Architecture - [Groupe GX] - [Early Stage | At Scale]

## 1. Contexte & contraintes
- Charge visée (QPS, volume de données, nombre d'utilisateurs)
- SLA / SLO (disponibilité, latence p50/p99, RPO/RTO)
- Contraintes non-fonctionnelles (budget, équipe, compliance)
- Hypothèses explicites

## 2. Diagramme d'architecture (C4 simplifié)
- Niveau 1 - System Context : qui interagit avec le système ?
- Niveau 2 - Containers : services, BDD, files, caches
- (Outil conseillé : draw.io, Excalidraw, Mermaid, ou schéma manuel scanné)

## 3. Choix techniques & alternatives rejetées (ADR-lite)
Pour chaque choix important, 3 lignes :
- **Décision :** ...
- **Alternatives considérées :** ...
- **Pourquoi ce choix :** ...

## 4. Flux de données critiques
- Débit d'un paiement (happy path) - en schéma + 5 lignes
- Gestion d'un échec / retry / timeout
- Idempotence : comment est-elle garantie ?

## 5. Points de rupture connus & seuils de bascule
- À quelle charge l'archi ne tient plus ?
- Quels signaux déclenchent la migration vers la version "At Scale" ?
  (ex : p99 > 500 ms, connexions PG > 80 %, stockage > 70 %)

## 6. Plan de migration  (uniquement dans le doc "At Scale")
- Étapes ordonnées pour passer de Early Stage à At Scale
- Ce qu'on peut faire sans downtime, ce qui en nécessite
- Ordre de grandeur de l'effort (personne-jours)

## 7. Risques & mitigations
Top 3 risques, chacun avec sa mitigation.

3.2 Ce qu'on attend concrètement

Document "Early Stage"

  • Objectif : faire tourner le système maintenant, avec l'équipe et les moyens actuels.
  • Ordre de grandeur : ≤ 50 QPS, < 1 M de lignes, 1 à 3 serveurs.
  • Choix typiques : un seul PostgreSQL, cron + script Python, dashboard BI simple, pas de Kafka.
  • Focus : simplicité, time-to-market, observabilité minimale.

Document "At Scale"

  • Objectif : supporter 5 M tx/mois, 500 k utilisateurs actifs, multi-datacenter.
  • Ordre de grandeur : > 500 QPS en pic, partitionnement/sharding, réplication, streaming.
  • Choix typiques : AWS RDS PostgreSQL Multi-AZ avec read replicas, AWS Kinesis ou SQS pour le streaming, RDS Proxy, materialized views, cache ElastiCache Redis.
  • Focus : scalabilité horizontale, résilience, cohérence (eventual vs strong), coût maîtrisé.
Règle d'or Pas de "cargo-culting". Si vous mettez Kafka dans votre archi "At Scale", expliquez pourquoi et quel problème précis il résout chez NAFAD-PAY. Un doc avec 15 briques AWS sans justification = note < la moyenne.

Règle commune à tous les groupes

Chaque doc d'archi doit obligatoirement traiter ces 4 dimensions :

  1. Hébergement & déploiement - tous les groupes travaillent sur AWS (région eu-west-3 Paris). Quel service précis, quels AZ, quelle topologie
  2. Réseau - VPC, sous-réseaux privés/publics, firewall, peering, pas d'IP publique inutile, diagramme réseau
  3. Sécurité - chiffrement at-rest et in-transit, gestion des secrets, IAM moindre privilège, protection DDoS/injection, threat model (top 3 attaques + mitigations)
  4. Protocole de communication - REST / gRPC / JDBC / SQS / Kinesis, pourquoi, quelle authentification, quel rate limiting

G1 - Core Banking Team : Base OLTP PostgreSQL PostgreSQL + Docker + AWS RDS

Rôle dans l'entreprise : Core Banking Backend Engineers. Vous concevez la base qui stocke chaque centime. Si elle tombe, perd ou corrompt une transaction, la fintech coule. Vous êtes l'équipe la plus proche de la comptabilité.

Implémentation (simple, ~5 jours de code)
  • docker-compose.yml avec un seul service : PostgreSQL 16
  • Modèle de données 3NF : users, accounts, transactions, merchants, agencies, agents, fees - avec contraintes FK, CHECK, UNIQUE
  • Index justifiés (un par requête critique identifiée, pas plus)
  • Script SQL de chargement des CSV fournis dans G1_OLTP/
  • Tests techniques : EXPLAIN ANALYZE sur les 5 requêtes les plus fréquentes, bench d'insertion concurrente (10 writers simultanés)
  • Tests métier : impossible de créer une transaction avec idempotency_key en doublon, solde jamais négatif (contrainte CHECK ou trigger), balance_after = balance_before ± amount vérifié
  • Stratégie à choisir pour les 43,8 % de références orphelines (zone quarantine ou rejet strict)
Doc d'architecture AWS (complexe, ~3 jours)
  • Hébergement : AWS RDS PostgreSQL 16 Multi-AZ en région eu-west-3 (Paris). Instance Graviton db.m7g pour le prix/perf. Comparer avec EC2 bare Postgres (moins cher, plus d'ops) et Hetzner (80 % moins cher mais pas managé)
  • Réseau : RDS dans un subnet privé, aucune IP publique. Accès uniquement via PrivateLink ou VPC peering depuis le VPC applicatif. Security Group qui autorise le port 5432 uniquement depuis le SG de l'API. Schéma réseau obligatoire
  • Sécurité :
    • Chiffrement at-rest avec AWS KMS (CMK gérée par NAFAD-PAY, rotation annuelle)
    • TLS 1.3 obligatoire, rds.force_ssl = 1
    • Mots de passe dans AWS Secrets Manager, rotation automatique 30 jours via Lambda de rotation
    • Comptes Postgres dédiés par service (app_writer, app_reader, etl_loader) avec GRANT minimal
    • Audit : pgaudit activé, logs exportés vers CloudWatch Logs avec rétention 90 jours
  • Threat model : que se passe-t-il si l'API backend est compromise (attaquant avec credentials app_writer) ? si un dev pousse un DROP TABLE ? si un snapshot S3 fuite ? Proposer 3 mitigations concrètes (IAM conditions, snapshots chiffrés, blast radius limité par rôle)
  • Protocole : PostgreSQL wire protocol sur TLS. Pooling via RDS Proxy (évite l'épuisement de connexions, failover transparent). Aucune connexion directe d'un dev en prod (passage par Session Manager pour le DBA)
  • Isolation & scale : stratégie de partitionnement natif PostgreSQL (par mois sur transaction_date), read replicas sur AZ différente, signal de passage à Aurora ou au sharding applicatif (quand > 5 k TPS en écriture)
  • Backup & DR : snapshots RDS automatiques + PITR 7 jours, copies cross-region (eu-west-1) pour DR, test de restore mensuel, RPO = 5 min, RTO = 30 min

G2 - Platform & API Engineering Team : API + mini UI React API + React + AWS ECS

Rôle dans l'entreprise : Platform / API Engineering Team. Vous exposez le système de paiement au monde extérieur. Chaque appel à votre API doit être traçable, sécurisable et scalable. Vous êtes la façade - et la première cible d'une attaque.

Implémentation (simple, ~5 jours de code)
  • API (Python/FastAPI ou Go) avec 4 endpoints :
    • POST /transactions - simule une transaction (écrit dans la DB de G1)
    • GET /transactions?limit=50 - liste les dernières transactions (pagination obligatoire)
    • GET /stats - agrégats simples (volume jour, taux de succès)
    • POST /simulate/batch?n=1000 - génère N transactions synthétiques
  • Mini interface React (Vite) sans auth : un formulaire "générer X transactions" + un tableau live des 50 dernières tx. Démonstration, pas produit fini
  • docker-compose.yml : api + react + postgres
  • Tests d'intégration (pytest/httpx ou Go testing)
Doc d'architecture AWS (complexe, ~3 jours)
  • Hébergement :
    • API sur AWS ECS Fargate derrière un ALB (task multi-AZ). Alternative comparée : Lambda derrière API Gateway (latence de cold-start à évaluer)
    • Front React bundle statique sur S3 + CloudFront CDN (cache à l'edge, HTTPS ACM)
  • Réseau :
    • ALB public dans subnets publics, tâches ECS dans subnets privés
    • ECS se connecte à RDS (G1) via le SG privé, port 5432 autorisé seulement depuis le SG ECS
    • Sortie Internet des tâches via NAT Gateway (ou VPC endpoints S3/Secrets Manager pour éviter NAT)
    • Schéma obligatoire : Internet → CloudFront → ALB → ECS → (SG privé) → RDS
  • Sécurité (conçue même si non implémentée) :
    • Auth : Cognito User Pools pour les end-users (OAuth2/OIDC), API keys via API Gateway usage plans pour partenaires B2B - justifier le choix
    • Rate limiting : AWS WAF rate-based rules par IP, AWS Shield Standard inclus, Shield Advanced si trafic sensible
    • WAF : règles managées AWS (CRS, Known Bad Inputs, SQL Injection, XSS)
    • Secrets : Secrets Manager, injection au boot de la tâche via IAM task role. Pas de .env en prod
    • HTTPS obligatoire, HSTS, CSP stricte, CORS whitelist sur le domaine React
    • Protection OWASP Top 10 (SQL injection via prepared statements même avec ORM, CSRF tokens, XSS sanitization, IDOR avec contrôle d'autorisation ligne par ligne)
  • Threat model : DDoS massif, fuite d'une API key (rotation + révocation), XSS sur le React (CSP nonce), replay attack (idempotency-key header obligatoire côté API). 3 mitigations par menace
  • Protocole : REST/JSON pour l'externe (compatibilité), HTTP/2, compression gzip, pagination obligatoire sur les GET liste. gRPC interne si microservices
  • Observabilité : logs JSON structurés → CloudWatch Logs, métriques p50/p95/p99 → CloudWatch Metrics, traces via AWS X-Ray, alarmes sur erreurs 5xx et latence
  • Scale : ECS Service Auto Scaling (target tracking sur CPU/requêtes), RDS Proxy pour ne pas épuiser les connexions, SQS devant les écritures si pic violent (decoupling producer/consumer)

G3 - Data Engineering Team : Medallion Architecture Bronze / Silver / Gold + AWS S3

Rôle dans l'entreprise : Data Engineering Team. Vous faites circuler et nettoyer les données entre les systèmes OLTP et analytiques. Un doublon, une ligne perdue, et la compta ne tombe plus juste. Vous êtes l'épine dorsale de la data.

Pourquoi Medallion Architecture ? C'est le standard de fait pour l'ingestion moderne (Databricks, Snowflake, AWS Lake Formation). Trois couches : Bronze (données brutes immuables), Silver (nettoyées, typées, dédupliquées), Gold (agrégats métier prêts pour G4). Chaque couche est reproductible à partir de la précédente - c'est la clé de l'idempotence.
Implémentation (simple, ~5 jours de code)
  • docker-compose.yml : postgres_warehouse + python_runner + minio (MinIO = S3 local pour dev)
  • Couche Bronze (ingestion brute) :
    • Lit les 7 dossiers daily_dumps/2024-01-XX/ et les copie dans bronze/ sur MinIO
    • Aucune transformation : AS-IS, même les lignes corrompues
    • Métadonnées ajoutées : _ingested_at, _source_file, _batch_id
    • Table technique bronze_runs qui journalise chaque ingestion
  • Couche Silver (nettoyage) :
    • Lit Bronze, dédoublonne par (reference, id), applique les soft-deletes du 19 janvier
    • Réconcilie full dumps (15, 18 janvier) vs deltas : le full du 18 est le point de contrôle
    • Normalise les schémas (G3 a source_node/processing_node/source_datacenter/processing_datacenter - les consolider)
    • Flagge les anomalies : _has_clock_skew (completed_at < created_at = 2 575 lignes détectées)
    • Charge dans PostgreSQL, tables silver_users, silver_accounts, silver_transactions, silver_fees
  • Couche Gold (agrégats métier) :
    • Tables prêtes pour G4 : gold_daily_volume, gold_user_activity, gold_agency_perf
    • Rafraîchissement incrémental, pas de full refresh
  • Test d'idempotence : rejouer Bronze→Silver→Gold deux fois doit donner un état bit-exact identique
  • Rapport d'anomalies (1 page Markdown) : 2 575 clock skews, 58 soft-deletes, volume cumulé vs full du 18
Doc d'architecture AWS (complexe, ~3 jours)
  • Hébergement cible :
    • Bronze : S3 avec partitionnement Hive (s3://nafad-bronze/dt=2024-01-15/), format Parquet ou CSV+manifest, Object Lock pour immutabilité
    • Silver : S3 en Parquet partitionné, catalogué via AWS Glue Data Catalog, requêtable via Athena
    • Gold : RDS PostgreSQL séparé du DWH de G4 (ou Athena si volumes lourds)
    • Orchestration : EventBridge scheduler + Step Functions qui enchaîne Bronze → Silver → Gold, chaque étape étant un ECS task Fargate
  • Source des dumps : bucket S3 dédié avec notifications EventBridge à l'arrivée d'un nouveau dump. Signed URLs si ingestion externe
  • Réseau : ECS tasks dans subnets privés, accès S3 via VPC Endpoint Gateway (pas de passage par Internet), accès RDS Gold via Security Group privé
  • Sécurité :
    • IAM role par étape Step Functions, scope minimal : Bronze task peut lire bucket source + écrire bucket Bronze. Rien de plus.
    • Chiffrement S3 par défaut (SSE-KMS avec CMK dédiée par couche Bronze/Silver/Gold)
    • Secrets DB dans Secrets Manager, injection via task role
    • Validation en amont : taille max du dump (10 GB), schéma CSV vérifié avec pandera ou great_expectations, quarantaine dans s3://nafad-quarantine/ si corrompu
  • Threat model : dump altéré en transit (checksum S3 + signature MFA Delete), zip bomb (limite de taille, extraction en streaming), injection de lignes malveillantes (validation schéma + sanity checks métier), credentials en logs (filtres CloudWatch + redaction)
  • Protocole : upload via S3 API HTTPS, chargement Postgres via COPY FROM sur connexion TLS, messages EventBridge signés
  • Observabilité : CloudWatch Logs structurés par étape Step Functions, Dead Letter Queue SQS pour les échecs, alarmes CloudWatch si un run échoue ou si le lag dépasse 15 min, métrique "lignes Bronze / lignes Silver / lignes rejetées"
  • Scale - passage au streaming : remplacer cron EventBridge par flux CDC (Debezium ou AWS DMS) depuis Postgres OLTP → Kinesis Data Streams → consumer ECS. Coexistence batch + streaming = Lambda Architecture (garder le batch pour le backfill, streaming pour le temps réel)

G4 - Analytics & BI Team : Data Warehouse & Dashboard SQL / dbt + outil BI + AWS Redshift

Rôle dans l'entreprise : Data / Analytics / BI Team. Vous fournissez les chiffres au DG et aux métiers. Un chiffre faux = décision stratégique fausse. Vous êtes aussi le dernier rempart de la gouvernance des données - vous gérez les PII des 10 000 utilisateurs.

Implémentation (simple, ~5 jours de code)
  • docker-compose.yml : postgres_dwh + outil BI au choix
  • Modèle en étoile : fact_transactions (grain = 1 ligne par tx) + dimensions dim_date, dim_user, dim_merchant, dim_agency, dim_node
  • Scripts SQL ou projet dbt-core pour transformer Gold (G3) → star schema
  • Dashboard BI (outil au choix) avec 5-6 des 15 questions métier (business_questions.md)
  • Choix SCD obligatoire : expliquer et justifier la stratégie retenue (Type 1, Type 2 ou hybride) pour dim_user
  • Stratégie à définir pour les 1 549 CONFLICT dont ~1 529 n'ont pas amount_node_a/b renseignés
  • Rapport sur l'écart de 1,3 M MRU entre node_metrics et stg_transactions
Doc d'architecture AWS (complexe, ~3 jours)
  • Hébergement à scale : AWS Redshift Serverless (pay-per-use, pas de gestion de capacité) OU Athena sur S3 Parquet si le volume reste modeste (< 1 TB). Comparer : Redshift pour requêtes interactives sub-seconde, Athena pour coût minimum et flexibilité
  • Séparation OLTP / DWH : le DWH ne doit jamais tourner sur la même instance que l'OLTP de G1. Une requête analytique lourde ne doit pas bloquer un paiement. Schéma de séparation obligatoire
  • Réseau : Redshift en Enhanced VPC Routing, outil BI en ECS Fargate privé (ou service managé équivalent) derrière ALB + Cognito authentifiant les utilisateurs internes
  • Sécurité & gouvernance :
    • Row-level security via Redshift RLS : un chef d'agence ne voit que sa wilaya
    • Column-level masking : masquer phone, nni, email pour les analystes juniors (Redshift Dynamic Data Masking)
    • Catalogage via AWS Glue Data Catalog + tagging PII avec Macie (détection automatique des données sensibles)
    • IAM users/roles par outil BI, jamais de compte partagé, Redshift Query Monitoring Rules
    • Export CSV limité en taille + audit log via CloudTrail
  • Threat model : analyste qui exfiltre un dump complet (quota + audit), token/session outil BI qui fuite (rotation + SSO), requête runaway qui coûte 500 $ (WLM queues + concurrency scaling limit). 3 mitigations concrètes
  • Protocole : Redshift via JDBC/ODBC TLS, outil BI ↔ Redshift via IAM Auth (IAM database credentials), accès utilisateur outil BI via SSO Cognito ou SAML
  • Scale & coût : materialized views Redshift pour les questions fréquentes, partitionnement/distribution par transaction_date, clustering par wilaya_id, ElastiCache Redis pour les compteurs live, Redshift Reserved Nodes si prévisible
  • Fraîcheur des données : batch J+1 pour les KPI historiques, near-real-time (toutes les 5 min) pour le jour courant, live counter via Kinesis → ElastiCache Redis pour le tx/seconde. Expliquer comment réconcilier les trois vues

Choix de stack par chemin critique

Dans une vraie fintech, le langage n'est pas un détail. Mais la vraie question n'est pas "quel est le meilleur langage", c'est "quel langage sur quel chemin". Un système de paiement a au moins trois chemins distincts avec des contraintes très différentes.

Les 3 chemins d'une fintech

API publique / BFF / gateway
(endpoints REST/GraphQL, webhooks)		 I/O-bound, throughput correct, time-to-market Go · Java · Node.js / TypeScript · Python (FastAPI) - tous valides ici

Pourquoi Go / Java / C# sur le write-path, et pas Python / Node ?

LangageForceLimite sur le write-path fintech
GoGoroutines légères, channels, compile en un binaire statique (déploiement trivial sur Hetzner), GC court et prédictibleÉcosystème métier bancaire moins riche que Java, moins de frameworks "clés en main"
Java / KotlinJVM mature, Project Loom (virtual threads) depuis Java 21, JTA, écosystème bancaire dominant (Revolut, N26, Adyen, Wise, Klarna tournent dessus)Image mémoire plus lourde, startup plus lent (atténué par GraalVM)

PythonRoi de la data (G2, G3, G4 !), ML, scripts d'intégration, backofficeLe GIL sérialise l'exécution Python : pas de concurrence CPU réelle sur un seul process. Typage dynamique = moins de garanties à la compilation sur de l'argent

L'erreur classique à ne pas faire Écrire le service de paiement en Python "parce qu'on connaît Python". Le GIL + le typage dynamique + l'absence de goroutines/virtual threads vont vous rattraper en production : timeouts, conditions de course, et potentiellement de l'argent perdu ou dupliqué. Python reste le bon choix pour les groupes G2, G3, G4 de ce projet (simulateur, ETL, DWH). Mais si un jour vous construisez le vrai write-path, passez à Go, Java ou C#.
Ce qu'on attend dans vos docs d'archi, par groupe
  • G1 (OLTP) : "Postgres pour le stockage. Pour l'application qui écrit dedans, Go ou Java/Spring Boot - pas Python - car le write-path est concurrent et chaque transaction doit respecter des invariants monétaires stricts."
  • G2 (Simulator) : "Python pour l'analyse et le simulateur batch (pandas, numpy, scipy). Si streaming temps réel, réécrire le producteur critique en Go."
  • G3 (ETL) : "Python + SQL pour le pipeline batch. En production streaming, l'étage critique de déduplication passe en Go ou Java, Python reste sur la transformation."
  • G4 (DWH) : "SQL + Python (dbt) - pas de débat, c'est l'écosystème standard."
Un étudiant qui raisonne par chemin et non par dogme de langage a compris le métier.

6. FAQ & règles du jeu

Q - Stack imposée ou libre ?

Stack conseillée : PostgreSQL / DuckDB / Python. Vous pouvez en proposer une autre si vous justifiez le choix dans votre doc d'archi Early Stage.

Q - Les 4 groupes utilisent-ils le même dataset ?

Non - chaque groupe a son propre jeu de données (volumes différents) généré avec la même seed 42 pour garder la cohérence métier (mêmes wilayas, mêmes types de tx, même périmètre). Vous n'avez pas besoin des livrables d'un autre groupe pour travailler.

Q - Taille des équipes ?

Chaque groupe est composé de 5 personnes. Organisation interne libre. Un seul dépôt Git, commits traçables par auteur.

Q - IA & assistants de code

Autorisés pour apprendre et débloquer, mais chaque choix d'architecture doit pouvoir être expliqué à l'oral. Un doc d'archi généré sans compréhension se repère en 30 secondes.

Q - Deadline & rendu

Lien du dépôt Git à envoyer au format :
[Nom Groupe - GX] - https://github.com/.../...
avec les deux docs d'archi en PDF ou Markdown à la racine.

Q - Anomalies dans les données : bug ou feature ?

Feature. Toutes les anomalies (soft-deletes, clock skew, conflits, doublons cross-DC, totaux node_metrics qui ne collent pas) sont intentionnelles. Les détecter et les gérer fait partie de la notation.
